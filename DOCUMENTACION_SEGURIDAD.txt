## 1. Anﾃ｡lisis del Caso: Amenazas Identificadas y Estrategia Seleccionada

### Amenazas Identificadas

| Amenaza | Descripci魹ｽn | Impacto |
|---------|-------------|---------|
| **Acceso no autorizado a endpoints** | Usuarios sin autenticaci魹ｽn acceden a recursos protegidos | Alto |
| **Escalaci魹ｽn de privilegios** | Usuario con rol limitado accede a funciones administrativas | Alto |

| **Fuga de informaci魹ｽn** | Exposici魹ｽn de datos sensibles como passwords o roles | Medio |

### Estrategia Seleccionada: JWT + BCrypt + Autorizaci魹ｽn por Roles

Se opt魹ｽ por un esquema de **autenticaci魹ｽn stateless basado en JWT** por las siguientes razones:

1. **Escalabilidad:** Al ser stateless, no requiere almacenamiento en servidor, facilitando horizontal scaling.
2. **Seguridad de credenciales:** BCrypt con factor de trabajo asegura que las contrase魹ｽas almacenadas son resistentes a ataques de fuerza bruta.
3. **Control de acceso granular:** Los roles (CLIENTE, EMPLEADO, ADMINISTRADOR) permiten aplicar el principio de m魹ｽnimo privilegio mediante `@PreAuthorize`.
4. **Estandarizaci魹ｽn:** JWT es un est魹ｽndar abierto (RFC 7519) con amplio soporte en la industria.

### 2.5 JwtAuthenticationFilter

Filtro que intercepta cada solicitud HTTP:
1. Extrae el token del header `Authorization: Bearer <token>`.
2. Valida el token usando `JwtUtil`.
3. Si es v魹ｽlido, carga el `UserDetails` y lo establece en el `SecurityContext`.
4. Si no hay token o es inv魹ｽlido, la solicitud contin魹ｽa (ser魹ｽ rechazada por el endpoint si requiere autenticaci魹ｽn).

## 3. Explicaci魹ｽn T魹ｽcnica

### 3.1 Protecci魹ｽn contra Acceso No Autorizado

**Mecanismo:** El `JwtAuthenticationFilter` se ejecuta antes de que cualquier solicitud llegue a los controladores. Verifica la presencia y validez del token JWT en el header `Authorization`. Sin un token v魹ｽlido, el `SecurityContext` queda vac魹ｽo y Spring Security rechaza la solicitud con HTTP 401.

**Flujo:**
```
Cliente -> Header: Bearer <token> -> JwtAuthenticationFilter -> SecurityContext -> Controller
```

### 3.2 Protecci魹ｽn contra Escalaci魹ｽn de Privilegios

**Mecanismo:** Se utiliza `@PreAuthorize` con expresiones como `hasRole('ADMINISTRADOR')` y `hasAnyRole('EMPLEADO', 'ADMINISTRADOR')`. Spring Security eval魹ｽa estos guards antes de ejecutar el m魹ｽtodo del controlador. Si el usuario autenticado no tiene el rol requerido, recibe HTTP 403 Forbidden.

**Ejemplos:**
```java
@PreAuthorize("hasRole('ADMINISTRADOR')")    // Solo administradores
@PreAuthorize("hasAnyRole('EMPLEADO', 'ADMINISTRADOR')")  // Empleados y admins
@PreAuthorize("hasAnyRole('CLIENTE', 'EMPLEADO', 'ADMINISTRADOR')")  // Todos autenticados
```

### 3.3 Protecci魹ｽn de Contrase魹ｽas (BCrypt)

**Mecanismo:** BCryptPasswordEncoder implementa el algoritmo bcrypt con salt incorporado. Cada contrase魹ｽa se almacena como un hash del formato `$2a$10$...` donde `10` es el factor de trabajo. Esto hace que:
- Dos usuarios con la misma contrase魹ｽa tengan hashes diferentes (por el salt).
- Ataques de rainbow table sean inviables.
- Ataques de fuerza bruta sean lentos (el factor de trabajo incrementa el tiempo de c魹ｽmputo).

### 3.4 Protecci魹ｽn contra Tokens Falsificados (JWT)

**Mecanismo:** Los tokens JWT est魹ｽn firmados con HMAC-SHA256 usando una clave secreta (`jwt.secret`). Sin conocer esta clave, un atacante no puede generar tokens v魹ｽlidos. `JwtUtil.parser().verifyWith(secretKey)` verifica la firma en cada solicitud.

### 3.5 Protecci魹ｽn contra CSRF

**Mecanismo:** Al usar `SessionCreationPolicy.STATELESS`, la aplicaci魹ｽn no crea sesiones en el servidor. Sin sesiones, los ataques CSRF (que dependen de cookies de sesi魹ｽn) no son aplicables. Por eso se deshabilita CSRF (`csrf.disable()`).

### 3.6 Protecci魹ｽn contra Intercepci魹ｽn

**Recomendaci魹ｽn:** En producci魹ｽn, toda la comunicaci魹ｽn debe hacerse exclusivamente por HTTPS para evitar que tokens y credenciales sean interceptados en tr魹ｽnsito (ataques Man-in-the-Middle).
